안녕하세요, 레인입니다 ☔️

이 글은 제가 한참 진행하다가 끝까지 가지 못한 프로젝트에 대한 회고입니다. 끝맺지 못한 게 아쉬워서, 어디까지 갔는지·무엇을 배웠는지라도 정리해두고 싶었어요.

주제는 한 줄로 요약하면 이렇습니다.

> **범위 내 기기를 검색하는 API의 꼬리 지연(tail latency)을 잡기 위해, MongoDB 2d 인덱스 기반 검색 구조를 S2 그리드 기반으로 바꾸려 했다.**

여기서 "꼬리"는 p95, p99 같은 응답 시간 분포의 꼬리 부분을 말합니다. 평균은 멀쩡한데 꼬리만 들쭉날쭉하면, 사용자 중 가장 느린 5%·1%의 경험이 망가지죠. 지연은 곧 매출 손실이라는 건 너무 유명한 이야기고요.

> Amazon → 100ms 지연마다 매출 1% 감소
> Google → 검색 결과가 0.5초 느려지면 트래픽 20% 감소
> 전자 거래 플랫폼 → 1ms 차이로 수백만 달러 손실 가능

## 1. 무엇이 문제였나

기기의 위경도 정보를 MongoDB에 저장하고, 클라이언트(지도 화면)가 보내준 bbox(bounding box) 범위로 기기들을 조회하는 흔한 패턴입니다.

```javascript
db.devices.find({
  geo_hash: {
    $geoWithin: {
      $box: [
        [lon_min, lat_min],
        [lon_max, lat_max]
      ]
    }
  },
  is_locked: true
})
```

문제는 같은 모양의 쿼리인데도 **실행 시점에 따라 응답 시간이 들쭉날쭉했다**는 점이었습니다. 어떤 때는 십수 밀리초에 끝나고, 어떤 때는 수백 밀리초가 걸리고. 배포한 것도 없고 인덱스를 바꾼 것도 없는데 `Query Targeting: Scanned Objects / Returned has gone above 1000` 알림이 한참 울리다가 어느 순간 잠잠해지곤 했죠.

### 원인 1 — 쿼리 플랜의 불안정성

MongoDB의 쿼리 플래너는 같은 Query Shape에 대해 여러 후보 플랜을 시험 실행하고 "가장 빠른 것 같은" 플랜을 캐시합니다. 그 후로는 캐시된 플랜을 그대로 씁니다.

프로파일러 로그에 다음과 같은 흔적이 자주 보였습니다.

```
"replanned": true,
"replanReason": "cached plan was less efficient than expected:
  expected trial execution to take 1402 works but it took at least 14020 works"
```

즉, **지오 인덱스와 non-geo 인덱스(예: `is_locked`)가 번갈아가며 캐시 플랜으로 채택**되고 있었습니다. 데이터 분포가 살짝 바뀔 때마다 플래너의 선택이 흔들리는 거죠.

### 원인 2 — 지오 인덱스의 낮은 selectivity

왜 플래너가 다른 인덱스를 고르려고 했을까요? **넓은 범위 조회에서 지오 인덱스의 선택도가 급격히 떨어지기** 때문입니다.

- Selectivity: 특정 조건이 전체 데이터 중 얼마나 좁은 범위를 골라내는지
- 좁은 bbox: 지오 인덱스가 적은 후보를 골라줌 → 선택도 높음 → 효율적
- 넓은 bbox: 지오 인덱스 후보가 폭증함 → 선택도 낮음 → 다른 인덱스가 더 효율적으로 보임

플래너 입장에서는 "전국 단위 bbox에서 지오 인덱스를 쓰면 후보가 너무 많으니, 차라리 `is_locked = true`(전체의 ~90%) 인덱스로 좁혀서 좌표는 CPU로 필터링하자"가 합리적인 선택처럼 보일 수 있습니다. 그런데 실제로는 **거의 풀스캔 + CPU 좌표 필터링** 폭증으로 최악의 결과가 나옵니다.

`hint`로 지오 인덱스를 강제할 수도 있지만, 구조적으로 인덱스 효율이 낮은 상황을 강제로 누르는 건 근본 해결이 아니죠.

## 2. 다른 서비스는 어떻게 풀고 있나

지리 검색은 우리만 겪는 문제가 아니라 거의 모든 지도/위치 기반 서비스가 같은 벽에 부딪힙니다. 들여다보니 큰 서비스들의 해법이 비슷했습니다.

- **Google Maps** → S2 Cell 기반 공간 인덱싱
- **Uber** → H3 (육각형 그리드)
- **Airbnb** → S2 기반 지역 셀 가공
- **배달앱/지도앱** → 타일 + 셀 조합

공통점은 **단일 DB의 지오쿼리에 의존하지 않고, 지구를 일정한 규칙으로 격자(grid)로 쪼개서 각 셀에 ID를 부여한다**는 점입니다. 그러면:

- 셀 크기(정밀도)를 우리가 조절할 수 있으니 **selectivity를 안정적으로 유지**할 수 있고,
- 조회 범위가 넓어져도 "겹치는 셀들"의 집합으로 표현되니 **인덱스를 일반 B-tree처럼 활용**할 수 있고,
- 쿼리 플래너가 잘못된 인덱스를 고를 여지가 줄어듭니다.

Uber도 이 길을 먼저 갔습니다.

> 검색 성능을 개선하기 위해 쿼리 크기를 줄이세요. Elasticsearch에서는 데이터 크기가 중요합니다. (…) 이 문제를 해결하기 위해 육각형 쿼리로 전환하여 지도를 육각형 셀로 나누었습니다.
>
> — [Uber Engineering 블로그](https://www.uber.com/en-KR/blog/elk/)

Elasticsearch든 MongoDB든, **range 기반 지오쿼리 → 셀 ID 기반 in-list 쿼리**로의 전환이라는 점에서는 같은 처방입니다.

## 3. GeoHash vs S2 vs H3

그리드 기술 후보를 비교했습니다.

### GeoHash

- 2008년 개발, 가장 오래되고 단순함
- 위경도 왜곡 존재 (위도에 따라 셀 크기 비대칭)
- 문자열 기반 해시, Z-order라 인접 셀의 ID가 인접하다는 보장이 없음

### S2 (Google, 2017)

- Hilbert Curve 기반 → **인접 셀의 ID가 서로 가깝다**
- 셀은 정사각형 기반 → 직관적이고 안정적
- **Cell ID = 64bit 정수** → B-tree 인덱스에 그대로 친화적
- Level 0 ~ 30의 명확한 계층 (낮을수록 큰 셀)
- Go 공식 라이브러리 `github.com/golang/geo/s2` 존재

### H3 (Uber, 2018)

- 육각형 기반 그리드
- 인접성 계산이 강력 (모든 이웃 셀과의 거리가 동일)
- 다만 CGO 필요, 부모-자식 구조가 정사각형만큼 깔끔하지 않음

저는 S2를 골랐습니다. 결정적이었던 건 다음 두 가지였어요.

1. **Cell ID가 64bit 정수**라서 MongoDB의 일반 인덱스에 그대로 태울 수 있다. (육각형은 ID 표현이 더 복잡해짐.)
2. **Go 라이브러리가 잘 정리되어 있다.**

## 4. 진짜 빨라지는지 벤치마크

이론은 그럴듯해 보였지만 실제로 빨라지는지 확인이 필요했습니다. 동일한 데이터셋(약 7만 건 규모)에 대해 같은 범위를 두 가지 방식으로 조회하고 비교했습니다.

- **AS-IS**: GeoHash + MongoDB 2d 인덱스 (`$geoWithin`)
- **신규**: S2 Cell ID 기반 인덱스 + 셀 커버링으로 후보를 좁힌 뒤 후처리 필터링

대한민국 내부에서 랜덤 좌표를 생성해 범위별로 1,000회·10,000회·100,000회씩 부하를 걸었습니다.

### 결과 동일 보장 (후처리 필터링 포함)

| 조회 범위 | GeoHash avg | S2 avg | 개선 | GeoHash p95 | S2 p95 | 개선 |
| --- | --- | --- | --- | --- | --- | --- |
| 좁은 (~1km)   | 13.09 ms  | 12.58 ms  | **3.9%**  | 15.28 ms  | 14.41 ms  | **5.7%**  |
| 중간 (~10km)  | 18.40 ms  | 16.36 ms  | **11.1%** | 39.20 ms  | 31.60 ms  | **19.4%** |
| 넓은 (~100km) | 345.77 ms | 195.09 ms | **43.6%** | 498.01 ms | 297.25 ms | **40.3%** |

### 후처리 필터링을 떼면 (오버페치 허용)

| 조회 범위 | GeoHash avg | S2 avg | 개선 | GeoHash p95 | S2 p95 | 개선 |
| --- | --- | --- | --- | --- | --- | --- |
| 좁은 (~1km)   | 32.49 ms  | 27.92 ms | **14.1%** | 104.66 ms | 97.99 ms  | **6.3%**  |
| 중간 (~10km)  | 19.66 ms  | 13.76 ms | **30.0%** | 42.25 ms  | 16.95 ms  | **59.9%** |
| 넓은 (~100km) | 351.44 ms | 66.62 ms | **81.0%** | 506.65 ms | 135.44 ms | **73.3%** |

읽고 나면 그림이 보입니다.

- **좁은 범위에서는 이득이 크지 않다.** 2d 인덱스도 좁은 범위에서는 충분히 잘 동작합니다.
- **중·대 범위에서 진가가 나온다.** 평균뿐 아니라 **p95가 같이 좋아진다**는 게 핵심입니다. 즉 **꼬리가 잘립니다.**
- **후처리 필터링을 빼면 더 빨라지지만 오버페치가 생긴다** — 셀 단위로 잘리니까 셀 경계 밖 좌표가 따라옵니다. 정확도와 성능의 trade-off는 명확합니다.

## 5. 정밀도(Cell Level) 결정 — 모니터링으로 풀기

S2에서 Cell Level은 0~30 사이의 정수이고, 레벨이 1 증가할 때마다 셀 면적이 4분의 1로 줄어듭니다. 정밀도 선택이 곧 성능을 좌우합니다.

- **셀이 너무 크면** → 한 셀에 너무 많은 기기가 들어가서 후처리 비용↑, 네트워크 낭비
- **셀이 너무 촘촘하면** → 커버링 셀 개수가 폭증해서 Mongo 쿼리의 `$in` 조건이 길어지고 인덱스 효율↓

S2 공식 통계 ([s2geometry.io](https://s2geometry.io/resources/s2cell_statistics.html))를 참고하면 대략 다음과 같습니다.

| Cell Level | 한 변 길이 (대략) |
| --- | --- |
| 9  | ~18 km   |
| 10 | ~9 km    |
| 11 | ~4 km    |
| 12 | ~2 km    |
| 13 | ~1.1 km  |
| 15 | ~280 m   |
| 16 | ~140 m   |

그래서 처음에는 zoom level별로 이런 매핑을 가설로 잡았습니다.

- 동네 범위 (~1km)   → Level 15
- 시/구 범위 (~10km) → Level 12
- 광역 범위 (~100km) → Level 9

그런데 "벤치마크로는 이렇게 나오는데 실제 트래픽은 어떨까?" 하는 질문이 남았습니다. 그래서 운영 환경에 모니터링을 붙였습니다.

기록한 데이터:

- `bbox_min`, `bbox_max` — 검색 범위 좌표
- `bbox_width_m`, `bbox_height_m`, `bbox_max_dim_m` — 범위 크기 (미터)
- `response_time_ms`, `result_count` — 응답 시간과 결과 수

그렇게 한참 쌓고 분석한 결과는 좀 의외였습니다.

> 범위 검색에서 응답시간 100ms 초과 트랜잭션은 **전체의 1% 이내**이고, 거의 대부분 **5km 이상 범위 검색**에 집중되어 있다.

즉, 일상 트래픽의 대부분은 사용자가 자기 주변(~1km) 또는 동네 단위(~수 km)를 보는 패턴이었습니다. tail의 원인은 명확히 **5km 이상의 광역 조회**였고요. 이 발견은 다음 단계의 의사결정을 바꿔놓습니다.

## 6. MongoDB에 S2 Cell ID 저장하기

설계의 핵심은 **어떤 타입으로 저장할 것인가** 였습니다.

S2 Cell ID는 **uint64**입니다. 그런데 MongoDB는 기본적으로 **signed int64만 지원**합니다. uint64 → int64로 그대로 형변환하면 2의 63승을 넘는 값에서 부호 비트가 뒤집혀 정렬·비교가 깨집니다.

선택지를 비교했어요.

### String

- 장점: 값 손실 없음
- 단점: 숫자 비교 불가, 인덱스 정렬·검색 효율 저하, 64bit 정수의 장점을 다 버림
- ❌ 채택 안 함

### Decimal128

- 장점: 매우 큰 정수 저장 가능
- 단점: 공간 사용량 큼, 불필요한 정밀도
- ❌ 채택 안 함

### BinData (Binary)

- uint64 값을 **8바이트 바이트 배열**로 변환해 저장
- 인덱스 정렬 시 바이트 순서 = 숫자 크기 순서
- 사실 MongoDB의 2d 인덱스도 내부적으로 binary로 다룬다
- 단점: Base64로 표시되어 사람 눈으로 읽기 어려움 (디버깅 불편하지만 기능 문제는 아님)
- ✅ **채택**

### Big Endian vs Little Endian

`uint64 → []byte` 변환 시 엔디안 지정이 필요합니다. 시스템마다 기본 엔디안이 다를 수 있고, 명시하지 않으면 환경별로 표현이 달라질 위험이 있습니다.

**Big Endian으로 통일**하면:

- 모든 환경에서 동일한 바이트 표현
- **바이트 사전 순서 = 숫자 크기 순서** → MongoDB BinData 인덱스 정렬을 그대로 활용
- 아키텍처 간 혼란 방지

Go 코드 스케치 (생략된 부분 많음):

```go
import (
    "encoding/binary"
    "github.com/golang/geo/s2"
    "go.mongodb.org/mongo-driver/bson/primitive"
)

func cellIDToBinData(c s2.CellID) primitive.Binary {
    buf := make([]byte, 8)
    binary.BigEndian.PutUint64(buf, uint64(c))
    return primitive.Binary{Subtype: 0x00, Data: buf}
}

func cellIDFromLatLng(lat, lng float64, level int) s2.CellID {
    ll := s2.LatLngFromDegrees(lat, lng)
    return s2.CellIDFromLatLng(ll).Parent(level)
}
```

### 정밀도가 다른 Cell ID를 함께 저장

여기가 또 하나의 디자인 포인트였습니다. **하나의 cell ID만 저장하면 범위 크기마다 효율이 달라집니다.** 그래서 같은 좌표에 대해 **여러 Level의 Cell ID를 동시에 저장**하기로 했습니다.

```javascript
{
  _id: ObjectId("..."),
  location: { lat: 37.5, lng: 127.0 },
  // S2 cell ID at multiple levels, stored as BinData (big-endian uint64)
  cell_id_narrow:  BinData(0, "..."), // level 15 - ~280m
  cell_id_medium:  BinData(0, "..."), // level 12 - ~2km
  cell_id_wide:    BinData(0, "..."), // level 9  - ~18km
  is_locked: true,
  // ...
}
```

각 필드에 인덱스를 따로 걸어두면, **조회 범위 크기에 따라 적절한 정밀도 필드를 골라 쿼리**할 수 있습니다. 좁은 범위는 narrow를, 광역은 wide를. 한 셀에 들어가는 후보 수와 커버링 셀 개수의 trade-off를 범위별로 다르게 운영하는 거죠.

## 7. 그런데 여기서 멈췄다 — 진짜 병목은 그 다음에 있었다

모니터링 데이터를 보면서 하나가 더 분명해졌습니다.

**넓은 범위 조회에서, 쿼리 자체는 빨라졌지만 `getMore` 단계가 새로운 병목이 되더라.**

MongoDB는 결과가 많으면 한 번에 다 안 주고 cursor로 나눠줍니다. 클라이언트가 `getMore`로 후속 배치를 받아오는데, 결과 개수가 수만 건이면 이 단계에서 들어가는 네트워크/디시리얼라이즈 비용이 커집니다. find 단계는 분명 줄었는데, **사용자 체감 응답 시간 기준으로는 개선이 크지 않은 구간**이 생긴다는 뜻이죠.

### 결국 진짜 질문

> "광역 조회에서 정말 모든 기기 정보가 다 필요한가?"

답은 명확합니다. 아니요.

- 사용자 대부분은 ~1km 이내에서 기기를 봅니다.
- 광역 화면에서는 수천~수만 개 기기를 점으로 다 찍어봐야 사용자가 개별 기기를 인식하지 못합니다. 화면 픽셀 수보다 마커가 더 많아지는 순간 정보로서의 가치가 사라져요.

### 다음 단계로 그렸던 그림

1. **Projection 최소화**: 광역 조회에서는 필요한 필드만 골라서 전송량 자체를 줄이기.
2. **클라이언트 클러스터링**: 광역에서는 개별 기기 상세 대신 셀별 count 같은 요약만 반환. 줌인하면서 좁은 범위가 될 때 상세를 조회. 마커 클러스터링 패턴.
3. **그리드 공유**: 서버의 cell ID 그리드와 클라이언트의 클러스터링 단위를 같은 기준으로 맞추면, 클라이언트가 "셀 단위로 병렬 호출"을 하는 것도 자연스럽게 가능.

즉 **서버 단독 최적화로는 끝나지 않는 문제**라는 결론이었고, 클라이언트와의 협업·API 재설계가 필요한 단계까지 갔습니다.

…그리고 거기서, 다른 우선순위에 밀려 멈췄습니다. 미완성. 🥲

## 8. 회고 — 끝까지 못 갔지만 남은 것

1. **"평균은 멀쩡한데 꼬리가 들쭉날쭉하다"는 신호를 가볍게 넘기지 말 것.**
   p95·p99의 불안정은 인프라/쿼리/플래너의 구조적 약점이 있다는 뜻이고, 평균치는 거기서 일어나는 일을 가려줍니다.

2. **쿼리 플래너에 의존하는 설계는 깨지기 쉽다.**
   같은 쿼리도 데이터 분포가 살짝 바뀌면 다른 인덱스로 점프합니다. "지금 잘 되니까 괜찮다"가 아니라, **데이터가 바뀌어도 같은 플랜이 나오는 구조**가 더 안정적입니다. 그리드 기반 검색이 그 답이 될 수 있고요.

3. **지오 인덱스의 선택도는 범위에 반비례한다.**
   좁은 범위에서는 잘 동작하지만 광역에서 무너집니다. 그리드를 깔면 selectivity 곡선이 평평해집니다.

4. **저장 타입 선택에서 uint64를 만나면 BinData(big-endian)를 떠올리자.**
   string·decimal128도 가능하지만, **바이트 정렬 = 숫자 정렬**이 되는 표현을 쓰면 인덱스를 그대로 살릴 수 있습니다.

5. **쿼리만 빨라진다고 사용자 체감이 빨라지진 않는다.**
   `getMore`, 네트워크 전송, 클라이언트 렌더링까지 같이 봐야 합니다. 광역 조회에서는 결국 "필요한 데이터만 주는" 설계 — 클러스터링, projection 최소화 — 로 넘어가야 합니다.

6. **미완성도 자산이다.**
   끝까지 못 간 게 아쉽지만, 다음에 비슷한 문제를 만나면 훨씬 빠르게 길이 보일 거라고 생각해요. 이번 회고도 그 일부고요.

언젠가 시간이 되면 클러스터링까지 마저 해보고 싶네요.
